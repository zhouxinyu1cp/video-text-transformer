### 一、 调整后技术栈推荐

#### 1. 前端技术栈
*   **核心框架**：React + TypeScript
*   **状态管理**：React Context API 或 Zustand
*   **UI组件库**：Ant Design 或 MUI (Material-UI)
*   **富文本与PDF**：`react-markdown` (预览) ， `html2canvas` + `jspdf` (生成下载)
*   **文件处理**：浏览器 `FileReader` API， 通过 `fetch` 或 `Axios` 与后端API交互

**说明**：前端技术栈保持不变，因其主要处理交互逻辑，与后端架构解耦。

#### 2. 后端技术栈 (单体服务)
*   **核心框架**：**Python FastAPI**
    *   **理由**：高性能异步框架，完美支持IO密集型操作（如下载视频、调用AI API）。自动生成交互式API文档，便于前后端联调。是构建此类数据处理服务的理想选择。
*   **任务队列与异步处理**：**FastAPI BackgroundTasks + 数据库状态追踪**
    *   **理由**：摒弃Redis和Celery。对于“提交任务-异步处理-查询结果”的模式：
        1.  **任务提交**：用户提交视频链接后，API立即生成一个唯一`task_id`并存入数据库，然后使用 `BackgroundTasks` 将一个处理函数加入后台执行队列，随即向客户端返回 `{ task_id, status: 'processing' }`。
        2.  **状态查询**：前端轮询如 `GET /task/{task_id}/status` 的接口，后端从数据库中读取该任务的最新状态（`processing`, `success`, `failed`）和进度返回。
        3.  **结果获取**：处理完成后，将生成的文稿、图片文件路径、结构化数据等信息更新到该`task_id`对应的数据库记录中。前端通过`task_id`查询并获取结果。
*   **数据存储**：**SQLite 数据库**
    *   **理由**：轻量、无需独立服务、零配置。作为单体应用，使用SQLite存储**任务元数据**是绝佳选择。
    *   **存储内容**：
        *   `tasks` 表：存储 `task_id`、`status`（状态）、`progress`（进度百分比）、`video_url`（原链接）、`result_data`（JSON格式的文稿、结构化信息）、`file_paths`（相关文件的服务器本地路径）、`created_at`、`finished_at`、`expires_at`（过期时间）。
*   **文件存储**：**服务器本地文件系统**
    *   **目录结构示例**：

        ```
        /app
          /data
            /tasks
              /{task_id}       # 按任务ID隔离
                /input         # 存放下载的临时视频/音频文件
                /output        # 存放最终产物：文稿(.txt/.docx)、关键帧图片
            /tmp              # 其他临时文件
        ```
        
    *   **文件服务**：通过FastAPI的 `StaticFiles` 模块，将特定目录（如`/app/data/tasks/{task_id}/output`）临时挂载为静态资源目录，为前端提供生成文件的**临时下载链接**。链接应设计为具有时效性或与任务会话绑定。
*   **音视频处理**：
    *   **工具**：`FFmpeg`（命令行调用，需预装于服务器系统）。
    *   **Python库**：`ffmpeg-python`（用于包装FFmpeg命令）、`aiofiles`（用于异步文件操作）。
*   **AI能力集成**：
    *   **语音转写(ASR)**：调用第三方云服务API（腾讯云、阿里云等）。
    *   **文本优化与信息提取**：调用大语言模型API（如GPT、文心一言、DeepSeek等）。
    *   **关键帧提取**：使用`OpenCV`（Python库）在本地实现场景切换检测。

### 二、 核心功能实现关键点说明

在单体+本地文件架构下，实现方式需做如下调整：

1.  **视频处理流程**：
    *   后端接收到任务，在后台任务中：
        1.  在`/data/tasks/{task_id}/input/`目录下，通过`FFmpeg`或`youtube-dl`等工具下载视频（需注意合规性）或提取音频。
        2.  调用ASR API，将音频文件上传或传递音频流，获得带时间戳的文本。
        3.  将原始文本送入LLM API，按照提示词进行口语转书面、润色、提取结构化信息。
        4.  使用`OpenCV`处理视频文件，提取关键帧图片，保存至`/data/tasks/{task_id}/output/`。
        5.  将优化后的文稿、结构化数据（JSON）和图片路径信息，更新到数据库`tasks`表的对应记录中，并将状态改为`success`。

2.  **会话级数据与清理**：
    *   这是**本地存储架构的核心挑战**。必须实现一个**资源清理机制**。
    *   每个任务记录都有`expires_at`字段（如创建后24小时）。
    *   需要部署一个**独立的定时清理进程**（如一个Python脚本，通过`cron`或系统定时任务调度）：
        *   定期扫描`tasks`表，删除所有`expires_at`时间已过期的记录。
        *   在删除数据库记录**前**，根据记录中的`file_paths`，递归删除对应的本地文件目录（`/data/tasks/{task_id}`）。
    *   此举确保满足“不持久化用户数据”的安全需求，并防止服务器磁盘被占满。

3.  **文件安全与访问控制**：
    *   **绝不**将用户提供的原始文件名直接用作服务器存储路径，应使用`task_id`或UUID重命名，防止路径遍历攻击。
    *   提供文件下载时，务必验证请求是否关联有效的、未过期的`task_id`。例如，下载路径可设计为 `/download/{task_id}/{filename}`， 后端在处理此请求时，需先校验`task_id`是否存在且未过期，再读取对应文件返回。

**总结**：
此方案构建了一个**完全自包含、易于部署**的后端服务。所有依赖（Python环境、SQLite文件、FFmpeg）均可打包在一个容器或部署在一台虚拟机中。**关键优势**是架构极简，运维成本低。**主要权衡**在于：

1.  **可扩展性**：难以水平扩展，更适合中小规模并发。
2.  **可靠性**：SQLite在高并发写场景下可能成为瓶颈，本地磁盘IO性能有限。
3.  **容灾**：服务器故障会导致所有进行中的任务丢失，且无内置数据冗余。

但对于项目早期、验证阶段或预期负载不高的场景，此调整后的技术栈是一个务实、高效且完全可行的选择。
